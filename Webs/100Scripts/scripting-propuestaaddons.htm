Scripting

Propuesta general para identificar addons
- Por Locorl

Propósito

La idea es que cuantos más scripts utilicen este u otro método concretado entre varios para identificar los addons mejor, yo propongo este porque me parece el más sencillo para poder identificar los addons. Este documento, es propenso a modificaciones... así que alguna que otra vez leetelo

Funcionamiento

Los addons que suelen distribuirse generalmente el primer archivo va con la extensión .mrc, aunque el sistema se usa para archivos de texto plano y de tipo ini, aunque sería válido igualmente para ambos, puesto que se inserta como un comentario, solo que en los inis hay que tener cuidado de no dañar ninguna sección.

Básicamente se trata de añadir un comentario en la pimera línea del addon con una serie de datos separados por un caracter poco frecuente (el resultante del $chr(127), que sale de la combinación control-retroceso), el invento quedaría así:

; Locorl  5.51  1.0b  http://dogscript.cjb.net/  dscom.txt  dscomleeme.txt   2  Addon para conectarse a otros servidores sin necesidad de usar otro mIRC

Uso el comentario que le tocaría a mi addon DsCom!, tengo bastantes más, pero este es el que me ha motivado a hacer primero un método que me siriviese en primera instancia a mí y luego para proponerlo como un método generalizado. Una pequeña aclaración, si en algún elemento no hace falta especificar algo se debe dejar un espacio en blanco, ya que el mIRC no distingue muy bien eso de dos caracteres juntos....

  • Pimer elemento:nombre del autor
  • Segundo: versión del mIRC que requiere el addon
  • Tercero la versión del addon, de mucha utilidad para el usuario que siempre quiere tener lo ultimo :-)
  • Cuarto: es la dirección de la web donde puede encontrarse
  • El quinto se refiere al archivo de ayuda, yo considero que debería ser una obligación por parte del autor incluir esto
  • Sexto: es el leeme donde se incluyen notas, aclaraciones...
  • Séptimo: aquí se debe poner qué archivos que se vayan a cargar dependen del addon
  • Octavo: Determina si es un alias o un remote, el 1 significa alias, el 2 remote
  • Noveno: La descripción del addon, demasiado larga como para aparecer en una línea, pero todavía no se me ha ocurrido nada para separarlo, puesto que el retorno de carro no sirve para esto xP
  • Décimo y siguientes (si existen): en este lo dejo para tu imaginación o necesidades, yo pondría en el décimo un comando para "desinstalar" el addon o para indicar las variables que usa (evitaría muchos conflictos)

Solo tiene un inconveniente, si hacemos alguna utilidad para escribir el comentario, el comentario no se guardará realmente si editamos el archivo con el editor del mIRC, se debería de recargar el addon y despues editarlo, así nos evitaríamos "inexplicables" situaciones de que no se ha guardado el comentario :-)

Alternativas que se quedaron atrás

Bien, como siempre uno siempre da vueltas a un sistema, sopesa sus pros y contras y decide, yo elegí este frente a otros sistemas porque me parecía el más practico, aquí los listo y opinad vosotros:

  • Archivos separados: con la descripcción del addon y demás... no es mal sistema, pero en el momento de la verdad me parece poco práctico, sobre todo a la hora de generar el archivo de descripcción, puesto que este puede ser erróneo o no ser el que realmente debería ir con el addon
  • Archivo único con las descripciones según el nombre de archivo: muy bonito pero enseguida podemos tener serios conflictos si diferentes addons tienen archivos con el mismo nombre
